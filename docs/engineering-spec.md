# agent-collab 总工程规范

本文档用于统一收敛 `agent-collab` 项目的工程规范，覆盖协作模式、代码编写、审批门控、Git 规则、测试与文档要求。

---

## 1. 目标

本规范的目标是让协作系统在实现、审查、提交、回退与复盘过程中保持一致性、可维护性和可扩展性。

核心目标：

- 默认先方案后执行
- 保持高内聚、低耦合
- 保持 review-before-merge
- 保证测试失败不会被静默带过
- 保证 Git 行为与审批状态一致
- 保证文档、代码、状态流同步演进

---

## 2. 语言与表达规范

- 智能体提示词使用简体中文
- 协作说明、技能说明、工程文档使用简体中文
- 代码命名遵循项目语言常规命名规范，不使用中文变量名
- Commit 建议使用中文格式：`<类型>: <简短描述>`

---

## 3. 协作流程规范

默认协作模式为 **模式 A：先方案后代码**。

### 模式定义

| 模式 | 说明 |
|---|---|
| A | 先方案后执行，适合复杂任务、新功能、跨模块改动 |
| B | 快速实现后审查，适合小修复、小范围优化 |
| C | 全流程分阶段审批，适合高风险操作 |

### 流程顺序（工作流 v2）

**需求 → 模式选择 → 方案/执行 → 审查 → Git 放行 → 合并 → 复盘**

不得跳过关键闸口直接进入 merge，不得在失败状态下继续推进。

推荐细化：

- 模式 A：需求 → 方案确认 → 实现 → 审查 → Git 放行 → 合并 → 复盘
- 模式 B：需求 → 快速实现 → 快速审查 → Git 放行 → 合并 → 复盘
- 模式 C：需求 → 分阶段确认 → 实现 → 阶段审查 → Git 放行 → 合并 → 复盘

---

## 4. 代码编写规范

### 4.1 最小侵入

- 优先在现有清晰边界内扩展
- 只修改需求范围内的代码
- 不夹带无关改动
- 不为当前任务顺手重写无关模块

### 4.2 高内聚、低耦合

- 一个模块尽量只承担一种核心职责
- 相关逻辑尽量集中在同一职责边界内
- 模块之间通过清晰、最小化接口协作
- 能独立演进、独立回退的逻辑尽量不要硬耦合在一起
- 新增功能时优先扩展边界清晰的模块，必要时先拆分职责再扩展

### 4.3 可维护性

- 优先可读性，再追求技巧性实现
- 明确处理边界条件与异常路径
- 不依赖隐式状态或魔法行为
- 避免为了省事堆砌复杂分支

### 4.4 注释规范

- 注释应当**必要且清晰**
- 注释优先解释“为什么这样做”，而不是简单复述代码
- 不保留过期注释，不写空洞注释
- 对特殊状态切换、门控规则、兼容性处理、回退逻辑，应补简短说明
- 如果代码已经难到必须大量解释，优先考虑优化结构与命名

---

## 5. 审批与门控规范

- 所有高风险动作必须受审批门控控制
- `pending_approvals` 未清理时，不应继续高风险动作
- 模式 C 的审批强度最高
- 未经过审查与放行，不得进入合并阶段
- 审批、状态和阶段必须与状态文件保持一致

状态文件核心字段：

- `mode`
- `phase`
- `approval_policy`
- `pending_approvals`

Git 相关约束字段：

- `git_main_branch_protection`
- `git_require_clean_checkout`
- `git_block_on_pending_approvals`
- `git_require_review_before_merge`

前置门控统一入口：`hooks/scripts/pre-flight-check.sh`

## 6. Git 工作规范

### 6.1 基本原则

- 先看 diff，再决定提交动作
- 小步提交，避免大杂烩
- 只提交需求范围内文件
- 默认不直接 push 到主分支
- 所有 Git 动作都要具备回退路径

### 6.2 分支规则

- 不直接在 `main/master` 上长期开发
- 每项任务尽量使用独立分支
- 高风险场景强制独立分支

### 6.3 提交规则

- 一个提交只表达一个清晰意图
- 测试尽量与实现一起提交
- 能独立回退的改动尽量独立提交
- 无关改动必须拆开

### 6.4 review-before-merge

- 未经审查通过，不得 merge
- 模式 C 下，未经过最终放行，不得 merge / push

### 6.5 测试失败处理

处理顺序：

1. 先定位原因
2. 能修则修，修后重测
3. 修不了则停止推进
4. 回退到用户并说明情况

禁止：

- 忽略失败继续提交
- 删除测试制造通过
- 隐瞒失败状态

---

## 7. 测试与验证规范

- 任何重要改动都应有验证动作
- 能自动化验证时，优先自动化
- 无法在当前环境验证时，必须提供：
  - 测试命令
  - 预期结果
  - 实际限制
  - 回退建议
- 测试失败不能被视为已完成

---

## 8. 文档同步规范

- 功能变更应同步更新相关文档
- 技能入口、状态字段、hooks 行为变化时，应同步更新说明
- 结构变化时应同步更新 README 目录与入口说明
- 新增规则应优先沉淀到总规范文档或对应专题文档

---

## 9. 角色职责规范

| 角色 | 核心职责 |
|---|---|
| 协调者 | 控流程、定阶段、管节奏 |
| 规划者 | 拆任务、标风险、给执行顺序 |
| 执行者 | 按范围实现、验证、汇报 |
| 审查者 | 审代码、审边界、审一致性 |
| 仲裁者 | 审批放行、阻断高风险动作 |
| 反思者 | 复盘流程、沉淀经验 |

必须遵守：

- 审查者不直接替执行者写代码
- 协调者不替代审查者做质量判断
- 仲裁者不模糊给结论
- 反思者不引入新的需求范围

---

## 10. 最终要求

本项目中的实现、测试、审查、Git 与文档更新，必须共同服从以下原则：

1. **先方案，后执行**
2. **先验证，后交付**
3. **先审查，后合并**
4. **高内聚、低耦合**
5. **注释必要且清晰**
6. **失败不隐藏，问题可回退**
