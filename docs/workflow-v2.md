# agent-collab 工作流 v2

这是 `agent-collab` 的精简工作流版本，目标是减少重复判断、统一状态源、缩短协作路径。

---

## 1. 设计目标

- 减少重复审批
- 只保留关键闸口
- 统一状态文件作为唯一状态源
- 让 Git、审批与协作流程同轨运行
- 让模式 B 更轻、模式 C 更严格但更清晰
- 让前置检查收敛为单一 pre-flight 链路

---

## 2. 核心流程

**需求 → 模式选择 → 方案/执行 → 审查 → Git 放行 → 合并 → 复盘**

### 关键闸口

1. **方案确认**：模式 A / C 必经
2. **实现完成**：执行结束后进入审查
3. **合并放行**：审查完成后才能进入 Git merge/push

---

## 3. 模式差异

### 模式 A

- 默认模式
- 先方案后执行
- 需要一次明确方案确认
- 审查后再进入 Git 放行

### 模式 B

- 快速模式
- 只保留最少必要检查
- 以实现与快速审查为主
- 不额外增加复杂审批

### 模式 C

- 高风险模式
- 每个关键阶段都需要确认
- 审批最严格，但判断点要清楚

---

## 4. 状态源统一

工作流 v2 只认可 `.claude/agent-collab.local.md` 作为状态源。

状态字段建议保持最小而清晰：

- `mode`
- `phase`
- `approval_policy`
- `pending_approvals`
- `git_*` 相关保护字段

---

## 5. Git 接入方式

Git 不再被视为独立流程，而是嵌入到工作流中：

- `commit`：在实现完成、审查前后进行整理
- `push`：在合并前或发布前进行放行检查
- `merge`：在审查通过后执行

Git 相关检查应优先通过技能入口与 hooks 完成，而不是让用户记住多个分散动作。

前置检查统一由 `hooks/scripts/pre-flight-check.sh` 执行，内部按顺序完成审批检查与 Git 检查，避免重复判断。

---

## 6. 角色职责精简

| 角色 | 主要职责 |
|---|---|
| 协调者 | 决定模式与阶段流转 |
| 规划者 | 只负责方案与风险拆解 |
| 执行者 | 只负责实现与自测 |
| 审查者 | 只负责质量审查 |
| 仲裁者 | 只负责高风险放行 |
| 反思者 | 只负责复盘 |

---

## 7. 优化原则

- 少判断点，保留关键闸口
- 少入口，统一状态
- 少重复，文档分层
- 少耦合，职责单一
