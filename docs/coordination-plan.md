# 协作系统方案

## 目标

为 OpenCode 构建一个可扩展的智能体协作系统，支持中文提示词、模式 A/B/C、审批门控与复盘机制。

## 设计原则

- 最小侵入
- 先方案后代码
- 审批前置
- Git 友好
- 结果可复盘
- 高内聚、低耦合

## 设计约束补充

### 高内聚

- 同一个模块只处理一种核心职责
- 技能文件聚焦单一入口能力，不混入无关流程
- hooks 脚本只负责门控、记录或校验，不承担复杂业务编排
- 状态脚本只维护状态读写与状态校验，不夹带额外流程逻辑

### 低耦合

- 智能体、技能、hooks、脚本之间通过明确状态字段协同，不直接相互写死依赖
- 新增 Git 规则、审批规则或协作模式时，优先新增独立文件或独立字段，避免改动现有核心流程
- 文档、技能入口与脚本实现保持松耦合：文档描述流程，脚本负责执行，状态文件负责传递约束
- 若两个模块必须联动，应通过最小接口对接，避免一处改动牵连多处重写

## 代码编写规范补充

### 高内聚、低耦合

- 一个文件、模块或脚本应尽量聚焦单一职责，避免同时承担多个无关职责
- 相关逻辑应尽量放在同一职责边界内，减少跨文件来回跳转与隐式依赖
- 模块之间通过清晰、最小化的接口协作，避免把内部实现细节暴露给外部
- 能独立演进、独立替换、独立回退的逻辑，不应被强耦合写在一起
- 新增功能时优先扩展现有边界清晰的模块，若现有模块职责已混杂，应拆分后再扩展
- 避免“为了省文件”把规划、状态、审批、Git 规则等不同职责堆在同一实现中

### 注释规范

- 注释必须是**必要的**，只在代码意图不够直观、存在约束、边界、风险或设计原因时补充
- 注释必须**清晰、准确、可维护**，解释“为什么这么做”优先于复述“代码正在做什么”
- 不写空洞注释，不写与代码内容重复的注释，不保留过期注释
- 对状态字段、门控规则、特殊分支、回退逻辑、兼容性处理，应优先补充简短清晰的说明
- 如果代码本身已经足够清晰，应优先优化命名和结构，而不是依赖大量注释弥补

## 角色

- 协调者：任务分配与节奏控制
- 规划者：拆解任务并输出执行计划
- 执行者：按范围实现变更
- 审查者：校验质量与边界
- 反思者：复盘总结
- 仲裁者：审批放行

## 模式

- A：先方案后代码
- B：直接实现后审查
- C：全流程分阶段审批
