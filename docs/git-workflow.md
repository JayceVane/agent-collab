# agent-collab Git 工作规范

本文档定义 `agent-collab` 协作系统中的 Git 工作方式，用于约束分支、提交、审查、回退与测试失败处理。

---

## 1. 目标

Git 在本插件中不是单独行为，而是协作流程的一部分。

目标：

- 保证改动可追踪
- 保证提交粒度清晰
- 保证审查先于合并
- 保证失败时可快速回退
- 保证模式 A / B / C 下的 Git 行为一致

---

## 2. 核心原则

1. **先看 diff，再决定动作**
2. **小步提交，避免一次提交过多无关改动**
3. **只提交当前需求范围内的文件**
4. **未经审查通过，不得合并**
5. **测试失败优先修复，无法修复则回退给用户**
6. **默认不直接 push 到主分支**
7. **所有 Git 动作都要能说明回退路径**
8. **通过合理拆分提交保持高内聚、低耦合**

### 2.1 与高内聚、低耦合的关系

- 一个提交只表达一个清晰意图，避免把多个无关改动耦合在一起
- 同一提交中的文件应尽量属于同一职责链路，例如“实现 + 对应测试”或“文档 + 对应说明更新”
- 不同模块的改动如果可以独立回退，就应拆分为不同提交
- 避免用一次大提交把本可独立演进的模块绑死

### 2.2 与注释规范的关系

- 提交中若引入了不直观的 Git 门控、状态切换或回退逻辑，应补充必要且清晰的注释
- 注释应解释规则存在的原因、阻断条件或风险来源，而不是简单重复命令本身
- 若某段实现必须依赖特定分支策略、审批前提或状态字段，应在关键位置保留短注释说明

---

## 3. 角色分工

| 角色 | Git 职责 |
|---|---|
| 协调者 | 决定何时进入提交、审查、回退阶段 |
| 规划者 | 在方案中标出分支策略、提交边界、回退点 |
| 执行者 | 产生代码改动，整理提交范围，准备提交说明 |
| 审查者 | 审查改动是否可提交、可合并 |
| 仲裁者 | 对高风险 Git 动作做放行或阻断 |
| 反思者 | 复盘提交粒度、回退成本、流程问题 |

**注意：**
- Git 不由审查者单独管理
- Git 主流程由协调者控制
- 审查者只决定“是否可进入合并阶段”

---

## 4. 分支规范

### 4.1 默认策略

- 不直接在主分支长期开发
- 每项任务应在独立工作分支完成
- 分支名应表达任务意图

### 4.2 推荐命名

```text
feat/<short-name>
fix/<short-name>
refactor/<short-name>
docs/<short-name>
chore/<short-name>
```

示例：

```text
feat/agent-collab-git-policy
fix/approval-hook-blocking
docs/install-guide-update
```

### 4.3 模式对应分支行为

| 模式 | 分支要求 |
|---|---|
| A | 强制建议使用独立分支 |
| B | 小改动可在当前任务分支继续，但不建议直改主分支 |
| C | 必须独立分支，且禁止跳过审查直接合并 |

---

## 5. 提交规范

### 5.1 提交原则

- 一次提交只做一类事
- 配套测试应尽量与实现同一提交
- 无关改动必须拆开
- 调试代码、临时文件、无关格式化不得混入

### 5.2 提交格式

统一使用中文提交建议格式：

```text
<类型>: <简短描述>
```

类型范围：

- `feat`
- `fix`
- `refactor`
- `docs`
- `test`
- `chore`

示例：

```text
feat: 增加协作模式选择技能
fix: 修正审批门控脚本阻断逻辑
docs: 补充插件安装与使用说明
```

### 5.3 提交前检查

提交前至少确认：

1. `git diff` 是否只包含需求范围内改动
2. 测试是否通过，或是否已明确失败原因
3. 是否存在待审批项
4. 是否有可回退方案

---

## 6. 审查与合并规范

### 6.1 review-before-merge

这是强规则：

- **未审查通过，不得合并**
- **模式 C 下，未经过仲裁者确认，不得结束流程**

### 6.2 模式对应审查强度

| 模式 | 审查要求 |
|---|---|
| A | 方案确认 + 代码审查 |
| B | 快速代码审查 |
| C | 分阶段审查 + 合并前最终放行 |

### 6.3 允许合并的前提

必须同时满足：

- 审查者给出“通过”
- 无未处理的高风险问题
- 测试通过或用户已确认接受风险
- 当前状态允许进入下一阶段

---

## 7. 测试失败处理

### 7.1 处理顺序

1. 先定位失败原因
2. 能修则修，修后重测
3. 修不了则停止推进
4. 回退到用户，说明情况

### 7.2 回退到用户时必须提供

- 失败命令
- 预期结果
- 实际结果
- 失败判定标准
- 建议的下一步

### 7.3 禁止行为

- 不得忽略失败继续提交
- 不得删除测试来制造“通过”
- 不得隐瞒失败状态

---

## 8. 回退策略

### 8.1 最低要求

每个任务在进入实现前，都应有至少一种回退方案：

- `git revert <commit>`
- 丢弃本地未提交改动
- 回到上一个稳定提交

### 8.2 推荐回退说明模板

```text
如果本次改动引发问题，可使用以下方式回退：
1. git revert <commit>
2. 重新运行验证命令
3. 确认状态文件与审批状态恢复正常
```

### 8.3 模式对应回退强度

| 模式 | 回退要求 |
|---|---|
| A | 提供提交级回退说明 |
| B | 至少提供未提交改动的撤销路径 |
| C | 必须提供明确、可执行的回退命令 |

---

## 9. 推荐 Git 操作节奏

### 模式 A

1. 新建分支
2. 输出方案
3. 用户确认
4. 实现变更
5. 自测
6. 审查
7. 生成提交建议
8. 用户确认后提交/合并

### 模式 B

1. 确认影响面小
2. 在任务分支实现最小改动
3. 自测
4. 快速审查
5. 生成提交建议

### 模式 C

1. 新建独立分支
2. 规划阶段审批
3. 执行阶段审批
4. 审查阶段审批
5. 合并前最终放行
6. 准备回退命令

---

## 10. 与插件状态文件的关系

Git 工作规范应与 `.claude/agent-collab.local.md` 协同：

- `mode`：决定 Git 审查强度
- `phase`：决定当前是否允许进入提交/结束阶段
- `approval_policy`：决定是否阻断高风险动作
- `pending_approvals`：未处理时不应继续高风险 Git 动作

另外，Git 规范已接入 hooks 与状态流：

- `git_main_branch_protection`：保护 `main/master` 不直接执行高风险 Git 动作
- `git_require_clean_checkout`：切换分支、merge、rebase、pull 前要求工作区干净
- `git_block_on_pending_approvals`：存在待审批项时阻断高风险 Git 动作
- `git_require_review_before_merge`：未到审查/完成阶段时阻断 merge/push

对应检查入口：

- `hooks/scripts/pre-git-check.sh`
- `scripts/mode-controller.sh git-check`

---

## 11. 建议检查清单

在准备提交前，建议按以下顺序检查：

```text
[ ] 当前模式是否正确
[ ] 当前 phase 是否允许继续
[ ] 改动是否仅包含需求范围
[ ] 测试是否通过
[ ] 是否已有审查结论
[ ] 是否具备回退方案
```

---

## 12. 最终要求

本插件中的 Git 行为必须服从以下顺序：

**方案 → 实现 → 测试 → 审查 → 放行 → 提交/合并 → 复盘**

不得跳过审查直接进入合并，不得带着已知失败结果继续推进。
